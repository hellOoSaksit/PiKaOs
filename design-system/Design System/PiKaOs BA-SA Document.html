A1 · BUSINESS ANALYSIS

ภาพรวมและวัตถุประสงค์ทางธุรกิจ

PiKaOs เป็นพื้นที่ทำงานแบบ "agent-ops" ที่ให้ผู้ใช้สร้าง บริหาร และติดตามทีม AI Agent ได้เหมือนบริหารทีมงานจริง โดยห่อหุ้มความซับซ้อนด้วยอุปมาแบบ RPG เบา ๆ (ห้อง · เควส · คลังความรู้) เพื่อให้เข้าใจง่ายและใช้งานเพลิน — แต่ความเป็นมืออาชีพมาก่อนเสมอ

ปัญหาที่แก้

  • การตั้งค่าและกำกับ AI Agent หลายตัวมักกระจัดกระจาย ไม่มีศูนย์รวม ไม่มีการควบคุมสิทธิ์ และติดตามต้นทุน (token) ได้ยาก
  • ทีมต้องการมอบหมายงานให้ Agent พร้อมกฎ/บริบทร่วม และตรวจสอบผลงานย้อนหลังได้
  • ความรู้ขององค์กรกระจาย ทำงานซ้ำ และ AI ตอบโดยไม่มีแหล่งอ้างอิง

เป้าหมายทางธุรกิจ

  • รวมศูนย์การจัดการ Agent — สร้าง Agent พร้อมบทบาท ทักษะ เครื่องมือ โมเดล และกฎ ในที่เดียว
  • มอบหมาย & ติดตามงาน — กระดานเควสแสดงสถานะ คิว ความคืบหน้า และบันทึกการทำงาน
  • ควบคุมและตรวจสอบได้ — RBAC ละเอียด โควตา token รายคน และ audit log
  • ความรู้ร่วม — คลังความรู้ (Codex) + ค้นคืนแบบมีอ้างอิง (Recall) ลดงานซ้ำและ hallucination
KPI ตัวอย่าง: งานที่สำเร็จต่อสัปดาห์ · อัตราความสำเร็จ · เวลาเฉลี่ยต่องาน · การใช้ token เทียบโควตา · จำนวน Agent ที่ใช้งานจริง
A2 · BUSINESS ANALYSIS

ขอบเขต (Scope)

อยู่ในขอบเขต

  • จัดการผู้ใช้ บทบาท และสิทธิ์ (RBAC)
  • สร้าง/แก้ไข AI Agent + โปรไฟล์ + การ์ดตัวละคร
  • ห้องทำงาน (World) แบบ top-down ปรับแต่งได้
  • กระดานเควส + บันทึกการทำงาน (.md)
  • คลังความรู้ (Codex) + ค้นคืน (Recall)
  • จัดการเครื่องมือ (MCP/LINE/Telegram/HTTP …) + การเชื่อมต่อ API
  • Workflows · โควตา token · audit log
  • หลายภาษา/รูปแบบคำศัพท์ (i18n)

อยู่นอกขอบเขต

  • การรันโมเดล AI จริงบน production (ปัจจุบัน mock/เดโม่บางส่วน)
  • ระบบชำระเงิน / บิลลิ่ง
  • การ deploy โครงสร้างพื้นฐาน / on-prem
  • SSO / ยืนยันตัวตนระดับองค์กร (ปัจจุบัน local)
  • แอปเนทีฟมือถือ (เป็นเว็บแอป responsive)

A5 · BUSINESS ANALYSIS

ความต้องการทางธุรกิจ (Business Requirements)

IDความต้องการความสำคัญ
BR-01ผู้ใช้สร้างและกำหนดค่า AI Agent ได้ครบ (บทบาท ทักษะ เครื่องมือ โมเดล กฎ เป้าหมาย)สูง
BR-02มอบหมายงาน/เควสให้ Agent และติดตามสถานะตั้งแต่เข้าคิวจนเสร็จสูง
BR-03ควบคุมการเข้าถึงด้วยบทบาทและสิทธิ์ระดับการกระทำ (RBAC) ปรับแต่งรายคนได้สูง
BR-04จำกัดและติดตามการใช้ token ด้วยโควตารายคน (รายสัปดาห์/เดือน)สูง
BR-05บันทึกการกระทำสำคัญทั้งหมดลง audit log ที่ตรวจสอบย้อนหลังได้กลาง
BR-06เก็บความรู้ร่วม (Codex) และค้นคืนแบบมีแหล่งอ้างอิง (Recall)กลาง
BR-07เชื่อมต่อเครื่องมือภายนอกผ่านชนิดมาตรฐาน (MCP/LINE/Telegram/HTTP/SQL/SMTP …)กลาง
BR-08รองรับหลายภาษาและรูปแบบคำศัพท์ สลับได้ทันทีโดยไม่กระทบข้อมูลต่ำ
BR-09จัดห้องทำงานเสมือน (World) เพื่อมองเห็นทีมและสถานะแบบรูปธรรมต่ำ

FR-D6ตั้งค่า — ธีม ภาษา/คำศัพท์ การเชื่อมต่อ API คลังคอมโพเนนต์ทุกคน
A7 · BUSINESS ANALYSIS

Use Cases / User Stories

UC-01

สร้าง AI Agent ใหม่

ผู้กระทำสมาชิก / ผู้จัดการ / แอดมิน (agent.create)
เงื่อนไขก่อนเข้าสู่ระบบและมีสิทธิ์สร้าง Agent
ขั้นตอนหลัก
  1. เปิด "เหล่า Agent" → กดสร้าง
  2. กรอกชื่อ บทบาท ตำแหน่ง เลือกการ์ดตัวละคร/คลาส
  3. เพิ่มทักษะ เครื่องมือ โมเดล กฎ และเป้าหมาย
  4. บันทึก — Agent ปรากฏในทีมและห้องทำงาน
ผลลัพธ์มี Agent ใหม่ผูกกับผู้สร้าง (owner) พร้อมรับเควส
UC-02

มอบหมายและรันเควส

ผู้กระทำผู้มีสิทธิ์ quest.run
ขั้นตอนหลัก
  1. เปิด "กระดานเควส" → สร้างเควส (ชื่อ รายละเอียด ผู้รับผิดชอบ)
  2. มอบหมายให้ Agent แล้วสั่งรัน
  3. ติดตามสถานะ: เข้าคิว → กำลังทำ → รอตรวจ → สำเร็จ/ล้มเหลว
ทางเลือกผู้ถูก deny quest.run จะไม่เห็นปุ่มรัน
ผลลัพธ์เควสมีบันทึกการทำงาน (.md) และนับรวมในสถิติ
UC-03

จัดการสิทธิ์ผู้ใช้ (override)

ผู้กระทำแอดมิน (user.manage / role.manage)
ขั้นตอนหลัก
  1. เปิด "จัดการผู้ใช้" → เลือกสมาชิก
  2. ปรับบทบาท หรือ grant/deny สิทธิ์รายข้อ
  3. ระบบคำนวณสิทธิ์: role + grant − deny (deny ชนะ)
ผลลัพธ์สิทธิ์ใหม่มีผลทันที + บันทึกลง audit log
UC-04

เพิ่มความรู้และค้นคืนแบบมีอ้างอิง

ผู้กระทำผู้มีสิทธิ์ codex.manage (เพิ่ม) · ทุกคน (ค้น)
ขั้นตอนหลัก
  1. เปิด Codex → เพิ่มบันทึก (หัวข้อ ประเภท เนื้อหา แท็ก)
  2. ที่ Recall พิมพ์คำถาม → ระบบคืนคำตอบพร้อมแหล่งอ้างอิง
ผลลัพธ์ความรู้ถูกค้นเจอและอ้างอิงได้ ลดงานซ้ำ
UC-05

เชื่อมต่อเครื่องมือภายนอก

ผู้กระทำผู้มีสิทธิ์ options.manage
ขั้นตอนหลัก
  1. เปิด "จัดการเครื่องมือ" → เลือกชนิด (MCP / LINE OA / Telegram / HTTP / SQL / SMTP / Webhook)
  2. กรอกฟิลด์ตามสเปกของชนิดนั้น (ค่าลับเก็บแบบ secret)
  3. บันทึก — เครื่องมือพร้อมผูกกับ Agent
ผลลัพธ์Agent ใช้เครื่องมือที่ตั้งค่าไว้ได้
A8 · BUSINESS ANALYSIS

กฎทางธุรกิจ (Business Rules)

IDกฎ
BRule-01Admin มีทุกสิทธิ์เสมอ — บทบาท admin ได้ permission ครบทุกข้อโดยไม่ต้องกำหนด
BRule-02Deny ชนะ Grant — การคำนวณสิทธิ์สุดท้าย = สิทธิ์ของบทบาท + grant รายคน − deny รายคน (deny มาก่อน)
BRule-03โควตา token — แต่ละผู้ใช้มีเพดานต่อรอบ (รายสัปดาห์/เดือน) ค่า null = ไม่จำกัด
BRule-04ระงับเมื่อเกินโควตาต่อเนื่อง — ผู้ใช้ที่ใช้ token เกินเพดานซ้ำอาจถูกตั้งสถานะ suspended
BRule-05Agent มีเจ้าของ — ทุก Agent ผูกกับผู้สร้าง; การแก้/ลบ Agent ของผู้อื่นต้องมี agent.edit.any/agent.delete.any
BRule-06บทบาทระบบลบไม่ได้ — 4 บทบาทมาตรฐาน (system:true) แก้ไขชุดสิทธิ์ได้แต่ลบไม่ได้
BRule-07ทุกการกระทำสำคัญถูกบันทึก — สร้าง/ระงับผู้ใช้ ปรับโควตา แก้บทบาท grant/deny ลง audit log พร้อมผู้กระทำและเวลา
BRule-08ค่าลับไม่แสดงเต็ม — API key / token เก็บและแสดงเฉพาะเลขท้าย (ฟิลด์ชนิด secret)
BRule-09เนื้อหา vs UI — ข้อความ UI ต้องมาจาก i18n key; เนื้อหาจริง (แชต, เอกสาร .md, seed) เป็นภาษาตามที่เขียน ไม่แปลอัตโนมัติ
S1 · SYSTEM ANALYSIS

สถาปัตยกรรมระบบ

เว็บแอป single-page ฝั่ง client ทั้งหมด — React + Vite, สเตตคงอยู่ใน localStorage ของเบราว์เซอร์ (ยังไม่มี backend จริง เหมาะกับเดโม่/ต้นแบบ)

เลเยอร์

  • UI Kit (src/components/ui/) — คอมโพเนนต์พื้นฐานที่ออกแบบเอง (ปุ่ม ฟอร์ม ดรอปดาวน์ โมดัล ฯลฯ) + ui-kit.css
  • App primitives (src/components/components.jsx) — Btn, Panel, PageHead, StatTile, Badge …
  • Screens (src/screens/) — หน้าจอทั้งหมด รวมคลังคอมโพเนนต์ (screens-library)
  • Logic / data (src/lib/) — i18n, characters (TOOL_TYPES), store, ui-modal, notify, users/RBAC
  • i18n data (src/data/i18n/) — ไฟล์ภาษา <lang>-<lexicon>.json สแกนอัตโนมัติ

การคงสถานะ (Persistence)

ข้อมูลผู้ใช้ บทบาท สิทธิ์ audit, Agent, ตัวเลือก, การตั้งค่าเครื่องมือ, ธีม/ภาษา และตำแหน่งห้อง ถูกเก็บแยก key ใน localStorage (เช่น guildos-users-v2, guildos.characters.v1) จึงคงอยู่ข้ามการรีเฟรช

หมายเหตุการ build: ไฟล์ PiKaOs App Preview.html ถูก generate จาก src/ ทั้งหมด (รวม CSS/JSX + inline i18n) — เป็น build artifact ห้ามแก้โดยตรง

การออกแบบ i18n

ระบบภาษาเป็น key-based ที่ data-driven ทั้งหมด — สลับภาษา/รูปแบบคำศัพท์ได้ทันทีโดยไม่กระทบข้อมูล

  • ไฟล์ภาษา 1 ไฟล์ = 1 ภาษา + 1 รูปแบบคำศัพท์ ที่ src/data/i18n/<lang>-<lexicon>.json (meta + translations) สแกนอัตโนมัติ
  • หน้าจอเรียก t("ns.key", { var }) เท่านั้น — ห้าม hardcode ข้อความ UI
  • Fallback 4 ชั้น: lang+style → lang+default-style → default-lang+style → default-lang+default-style → ตัวคีย์เอง · default = en-formal
ไฟล์ภาษารูปแบบคำศัพท์หมายเหตุ
en-formalEnglishFormalMaster · default
th-formalไทยทางการฐานภาษาไทย
th-fantasyไทยแฟนตาซีนักผจญภัย/เควส/มานา
en-adventurerEnglishAdventureradventurer/quest/mana
th-wuxiaไทยจอมยุทธ์จอมยุทธ์/ภารกิจ/ลมปราณ
เนื้อหา vs UI: เฉพาะข้อความ UI ที่ใช้ key; เนื้อหาจริง (แชต AI, เอกสาร .md, seed data) เป็นภาษาตามที่เขียนไว้ ไม่ผ่านระบบแปล
